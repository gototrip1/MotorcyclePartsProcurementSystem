# 摩托车零部件采购管理系统 —— 详细设计文档

> 版本：1.0.0　|　日期：2026-06-16
> 系统定位：纯内部进销存系统（**采购入库 + 领用出库**，无对外销售）
> 核心原则：**库存的任何增减，都必须由管理员审核通过后才触发（职责分离）**
> 配套文档：[CLAUDE.md](../CLAUDE.md)、[功能列表](功能列表.md)、[数据库表结构](database_schema.sql)、[数据库表说明](数据库表说明.doc)、[顶层数据流图](顶层数据流图.drawio)、[0层数据流图](0层数据流图.drawio)、[系统功能图](系统功能图.drawio)

## 一、概述

### 1.1 编写目的
本文档面向开发、测试与维护人员，详细描述摩托车零部件采购管理系统的总体架构、角色权限、功能模块设计、数据库设计、数据流、关键业务流程与状态流转，作为编码实现与验收的依据。

### 1.2 系统简介
本系统管理摩托车零部件从「向供应商采购入库」到「内部车间领用出库」的全过程。系统不对外销售，属于企业内部进销存。整套设计围绕一条铁律展开：

> **创建单据 ≠ 改库存。** 任何库存的增加或减少，都只能由【系统管理员】审核通过后触发。采购员无权直接入库、领用人员无权直接出库——这正是「审核」这一步存在的意义。

### 1.3 设计目标
- **职责分离**：建单与改库存彻底分离，库存数据只读保护。
- **流程清晰**：采购入库、领用出库两条主线各自独立成表、互不污染。
- **可追溯**：单据状态机驱动，所有出入库可回溯到审核动作。
- **前后端分离**：后端 REST 接口 + Vue 3 前端，按角色做菜单/按钮级权限。

## 二、系统总体设计

### 2.1 技术栈

| 层 | 选型 |
|---|---|
| 后端框架 | Spring Boot 3.1.10 / Java 17 |
| ORM | MyBatis Plus 3.5.3.1 |
| 数据库 | MySQL 8.0.33（库名 `motorparts`） |
| 连接池 | HikariCP |
| API 文档 | SpringDoc OpenAPI（Swagger UI） |
| 工具 | Lombok、Hutool、FastJSON、Spring Validation |
| 前端框架 | Vue 3（Composition API） |
| UI 组件库 | Element Plus |
| 构建工具 | Vite |
| 通信 | axios 调用后端 `/api/**` |

### 2.2 后端分层架构（包路径 `com.motorparts`）

```
controller/   REST 接口，@RestController + @RequestMapping("/api/xxx")，构造器注入
service/      业务接口
service/impl/ 业务实现（库存变动等事务逻辑在此）
mapper/       MyBatis Plus Mapper（继承 BaseMapper）
entity/       实体，继承 BaseEntity（含 deleted / create_time / update_time）
dto/          查询/组合返回对象
common/       Result、PageResult、enums（角色/状态等）、exception（全局异常）
config/       MyBatisPlus、Swagger、WebMvc(CORS)、DDL 初始化
init/         DatabaseInitializer 启动建表/初始化数据
```

### 2.3 关键约定
- **统一响应**：`Result<T>{code, message, data, timestamp}`；分页用 `PageResult`。成功 `Result.success(data)`，失败 `Result.error(...)`。
- **接口前缀**：统一 `/api/`，RESTful 风格（如 `/api/parts/create`）。
- **跨域**：`WebMvcConfig` 对 `/**` 全量放开，便于前后端分离开发。
- **校验**：`@Validated` + `@Valid`，异常由 `GlobalExceptionHandler` 统一兜底。
- **枚举集中**：`common/enums`（角色、订单状态、零件分类等）。
- **库存只读保护**：库存数量字段不开放任何「直接赋值」接口，只能经审核出入库的事务逻辑变更。

## 三、角色与权限设计

### 3.1 三种角色

| 角色 | 角色码 | 定位 | 能做 | 不能做 |
|---|---|---|---|---|
| 采购员 | `purchase` | 只管进货 | 建档（供应商/零件）、开采购单、录物流、提交入库、标记付款 | 无权入库、无权领用、不能直接改库存 |
| 领用人员 | `requisition` | 车间/生产领料 | 开领用单、提交出库申请 | 不能直接扣库存、不参与采购 |
| 管理员 | `admin` | 审核中枢 | 审核档案/入库/出库、库存增减、用户管理、参数配置、报表 | —— |

### 3.2 功能 × 角色 权限矩阵

| 功能 | 采购员 | 领用人员 | 管理员 |
|---|:---:|:---:|:---:|
| 登录 / 退出 | ✅ | ✅ | ✅ |
| 新建/编辑 供应商、零件 | ✅(提交) | —— | ✅ |
| 审核供应商/零件、锁定单价 | —— | —— | ✅ |
| 创建采购单 / 录物流 / 提交入库 | ✅ | —— | —— |
| 标记已付款 | ✅ | —— | —— |
| 创建领用单 / 提交出库 | —— | ✅ | —— |
| 审核采购入库（库存 +） | —— | —— | ✅ |
| 审核领用出库（库存 −） | —— | —— | ✅ |
| 库存查询 / 预警查看 | ✅(只读) | ✅(只读) | ✅ |
| 库存预警参数配置 | —— | —— | ✅ |
| 用户管理 | —— | —— | ✅ |
| 统计报表 | 部分 | 部分 | ✅ |

> ✅ = 可操作；✅(提交) = 仅能发起需审核；✅(只读) = 仅查看；—— = 无权限。

## 四、数据流设计（DFD）

### 4.1 顶层数据流图（上下文图）
见 [顶层数据流图.drawio](顶层数据流图.drawio)。系统作为单一处理中心，三种角色作为外部实体：

- **输入（角色 → 系统）**
  - 管理员：登录 / 审核档案·入库·出库 / 用户管理 / 库存参数 / 报表查询
  - 采购员：登录 / 供应商·零件建档 / 创建采购单·录物流 / 提交入库 / 标记付款
  - 领用人员：登录 / 创建领用单 / 提交出库申请 / 查询领用状态
- **输出（系统 → 角色）**
  - 管理员：token+角色 / 各类数据 / 待办审核列表 / 统计报表
  - 采购员：token+角色 / 采购单状态 / 入库审核结果 / 库存(只读)
  - 领用人员：token+角色 / 领用单状态 / 出库审核结果 / 库存(只读)

### 4.2 0 层数据流图
见 [0层数据流图.drawio](0层数据流图.drawio)。系统拆分为 6 个加工过程，并连接对应数据存储：

| 加工 | 输入 → 输出 | 数据存储 |
|---|---|---|
| P1 登录认证 | 全部用户 → 全部用户 | D1 用户表（读：校验账号密码） |
| P2 用户管理 | 管理员 → 管理员 | D1 用户表（读/写） |
| P3 供应商/零件档案管理 | 采购员(提交)·管理员(审核) → 采购员·管理员 | D2 供应商表 · D3 零件表（读/写） |
| P4 采购入库管理 | 采购员 → 采购员 | D4 采购单/明细（读/写）+ D3 零件表（读单价） |
| P5 领用出库管理 | 领用人员 → 领用人员 | D6 领用单/明细（读/写） |
| P6 审核中心（库存闸门） | 管理员 → 管理员·采购员·领用人员 | D4·D6（读/改状态）· **D5 库存表（写 +/−）** |

> 关键点：只有 **P6 审核中心** 拥有写 D5 库存表的权限，其余加工都不触碰库存数量。

## 六、关键业务流程与状态机

### 6.1 采购单状态流转

| 状态值 | 状态 | 触发者 | 含义 |
|:---:|---|---|---|
| 1 | 采购中 | 采购员建单 | 起始态，自行处理物流/发货/等货，无需审核 |
| 2 | 待入库审核 | 采购员（货到提交） | 球交给管理员 |
| 3 | 已入库 | 管理员审核通过 | **库存 +** |
| 4 | 已取消 | —— | —— |

> 付款状态用独立字段 `paid`（0-未付款 / 1-已付款）标记，**不混入** `status` 状态机。
> 到货时间不由系统计算：`expected_delivery_date` 下单时约定填入，`actual_delivery_date` 货到时人工确认。

**采购入库流程**：
```
采购员创建采购单(采购中) → 录物流/标记发货 → 货到提交入库申请(待入库审核)
   → 管理员审核 → 通过(已入库, 库存+) / 驳回(退回原因)
   → 采购员对账后标记已付款(paid=1)
```

### 6.2 领用单状态流转

| 状态值 | 状态 | 触发者 | 含义 |
|:---:|---|---|---|
| 1 | 待审核 | 领用人员提交 | 球在管理员手上 |
| 2 | 已出库 | 管理员审核通过 | **库存 −** |
| 3 | 已取消 | —— | —— |

**领用出库流程**：
```
领用人员创建领用单 → 提交出库申请(待审核)
   → 管理员审核 → 先验证库存充足
        → 充足且通过(已出库, 库存−)
        → 库存不足或驳回(退回原因, 库存不变)
```

### 6.3 库存变动规则（核心铁律）
1. `inventory.current_quantity` 只能由**管理员审核通过**的入库（采购单 → 已入库）/ 出库（领用单 → 已出库）在事务中自动变更。
2. 出库审核必须**先校验库存充足**，不足则驳回。
3. 任何角色、任何接口都不能对库存数量做「直接赋值」。
4. 入库同时更新 `last_inbound_time`，出库同时更新 `last_outbound_time`。

## 七、数据库设计

数据库名 `motorparts`，字符集 `utf8mb4`，引擎 InnoDB。共 **8 张表** + **3 个视图**。所有表均含逻辑删除 `deleted` 与 `create_time / update_time`。详见 [database_schema.sql](database_schema.sql)、[数据库表说明.doc](数据库表说明.doc)。

### 7.1 表清单

| 表名 | 中文名 | 作用 |
|---|---|---|
| `user` | 用户/员工表 | 登录账号，按角色区分权限 |
| `supplier` | 供应商表 | 零件来源单位 |
| `part` | 零部件表 | 零件主数据，单价由管理员锁定 |
| `purchase_order` | 采购订单表（含物流） | 采购入库主单，物流字段已并入 |
| `order_detail` | 订单明细表 | 采购单逐行零件明细 |
| `inventory` | 库存表 | 每个零件一条库存记录 |
| `stock_out_order` | 领用单（出库单） | 领用出库主单 |
| `stock_out_detail` | 领用明细表 | 领用单逐行零件明细 |

### 7.2 表关系

```
user(采购员)   1 ── N  purchase_order   1 ── N  order_detail   N ── 1  part
user(领用人)   1 ── N  stock_out_order  1 ── N  stock_out_detail N ── 1 part
supplier       1 ── N  part             1 ── 1  inventory
```

### 7.3 关键字段约束
- `order_detail.subtotal` = `quantity × unit_price`，**数据库自动生成列（STORED）**，无需插入。
- `inventory.part_id` 唯一（保证 part 1—1 inventory）。
- 外键策略：`order_detail.order_id` → `purchase_order` 为 `CASCADE`（删单连带删明细）；`*.part_id` → `part` 为 `RESTRICT`（零件被引用则不可删）；`part.supplier_id` → `supplier` 为 `SET NULL`。

### 7.4 视图

| 视图 | 作用 |
|---|---|
| `inventory_warning_view` | 库存预警：按当前数量与预警值/安全库存比较，输出 严重不足 / 不足 / 充足 |
| `order_statistics_view` | 采购统计：按下单日期统计订单数、总金额、均价 |
| `order_detail_full_view` | 订单明细全信息：联零件与供应商的宽表 |

## 八、接口设计概要

接口统一前缀 `/api/`，RESTful 风格，统一响应 `Result<T>`。下表为主要资源接口（示例，具体以 Swagger UI 为准）：

| 模块 | 方法 | 路径 | 说明 | 权限 |
|---|---|---|---|---|
| 登录 | POST | `/api/auth/login` | 登录获取 token | 全部 |
| 登录 | POST | `/api/auth/logout` | 退出 | 全部 |
| 供应商 | POST | `/api/suppliers/create` | 新建（提交审核） | 采购员 |
| 供应商 | POST | `/api/suppliers/audit` | 审核通过/驳回 | 管理员 |
| 供应商 | GET | `/api/suppliers/page` | 分页查询 | 全部 |
| 零件 | POST | `/api/parts/create` | 新建（提交审核） | 采购员 |
| 零件 | POST | `/api/parts/audit` | 审核 + 锁定单价 | 管理员 |
| 采购单 | POST | `/api/purchase-orders/create` | 创建采购单 | 采购员 |
| 采购单 | POST | `/api/purchase-orders/submit-inbound` | 提交入库申请 | 采购员 |
| 采购单 | POST | `/api/purchase-orders/mark-paid` | 标记已付款 | 采购员 |
| 领用单 | POST | `/api/stock-out-orders/create` | 创建领用单 | 领用人员 |
| 领用单 | POST | `/api/stock-out-orders/submit` | 提交出库申请 | 领用人员 |
| 审核 | POST | `/api/audit/inbound` | 审核入库（库存+） | 管理员 |
| 审核 | POST | `/api/audit/outbound` | 审核出库（库存−） | 管理员 |
| 库存 | GET | `/api/inventory/page` | 库存查询 | 全部(只读) |
| 库存 | GET | `/api/inventory/warning` | 库存预警 | 全部 |
| 用户 | POST | `/api/users/create` | 新建用户 | 管理员 |
| 统计 | GET | `/api/statistics/purchase` | 采购统计 | 管理员 |

> 启动后访问 `http://localhost:8080/swagger-ui.html` 调试完整接口。

## 九、前端设计要点

- 独立前端工程（`frontend/`），npm 管理依赖，Vite 启动开发服务器。
- 页面用 Element Plus 组件搭建（表格、表单、弹窗、菜单等）。
- axios 统一封装请求，统一处理 `Result{code,message,data}`。
- **按 3 种角色做菜单/按钮级权限控制**：
  - 采购员：供应商、零件、采购单模块。
  - 领用人员：领用单模块。
  - 管理员：审核中心、用户管理、库存参数、统计报表及全部模块。
- 复用后端状态枚举（采购单/领用单状态）做前端流转展示与按钮可用性控制。
- 开发期 axios 直连本地 8080（后端已全量放开 CORS）。

常用命令：

```bash
npm install        # 安装依赖
npm run dev        # 启动开发服务器（Vite）
npm run build      # 构建生产产物
```

## 十、非功能性设计

| 方面 | 设计 |
|---|---|
| 安全 | 密码加密存储；按角色做接口权限校验；库存只读保护 |

| 可维护 | 分层架构、统一响应、全局异常处理、枚举集中管理 |
